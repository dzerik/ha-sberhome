# Правовая позиция проекта

## Что делает проект

`ha-sberhome` — Home Assistant integration, которая позволяет
**конечному пользователю** управлять **собственными устройствами**,
зарегистрированными в его учётной записи Sber Smart Home («Салют!»).
Интеграция работает строго **под управлением пользователя и от его имени**:

- авторизация — собственными учётными данными пользователя
  (Sber ID OAuth через `id.sber.ru` либо SMS-OTP на телефон, привязанный
  к аккаунту пользователя);
- запросы идут с IP пользователя, с его access_token-ом, к публично
  доступному gateway-серверу `gateway.iot.sberdevices.ru`;
- никаких чужих данных, токенов, credentials проект не использует и
  не запрашивает;
- никакого массового scraping'а, прокси-пулов, DoS-нагрузки на
  Sber-серверы интеграция не создаёт.

## Два пути авторизации

### 1. Sber ID OAuth — стандартный путь (документирован Sber)

Использует **публично задокументированные** endpoint'ы:

- `GET https://id.sber.ru/CSAFront/oidc/authorize.do` — OAuth 2.0 / OIDC
  authorization code flow с PKCE.
- `POST https://online.sberbank.ru:4431/CSAFront/api/service/oidc/v3/token`
  — token exchange.

Это **тот же flow**, что использует [официальный Sber ID iOS SDK](https://github.com/SberID/ios-sdk),
[Sber ID JS SDK](https://github.com/SberID/js-sdk) и публично документирован
на developer.sber.ru:

- [Sber ID — параметры запроса](https://api.developer.sber.ru/product/SberbankID/doc/v1/reqparametrs)
- [Sber ID — мобильные параметры](https://developer.sberbank.ru/doc/v1/sberbank-id/reqmobile)
- [Cloud migration checklist](https://developers.sber.ru/docs/ru/sberid/faq/a4-switching-to-cloud)

Этот путь — стандартная партнёрская интеграция через Sber ID, и
правовых вопросов к нему быть не должно.

### 2. SMS-OTP (beta) — путь на основе открытого источника

Альтернативный flow для пользователей, у которых стандартный путь не
работает (блокировка `companionapp://` в браузере, баги мобильного
UA-сниффинга и т.п.):

- `POST /CSAFront/uapi/v2/authenticate` (SMS init);
- `POST /CSAFront/uapi/v2/verify` (OTP → authcode);
- `POST /CSAFront/api/service/oidc/v3/token` (authcode → токены);
- `GET mp-prom.salutehome.ru/v13/smarthome/token` (SmartHomeToken).

Wire-format этих endpoint'ов получен **полностью из открытого
проекта** [shuryak/sberdevices](https://github.com/shuryak/sberdevices)
(Go, MIT-license, публичный с 2023 года). Все endpoint'ы, заголовки,
anti-bot `rsa_data` ритуал, persistent `X-Device-ID`, refresh-rotation
— переиспользованы оттуда без изменений по существу; здесь они
портированы на Python/asyncio.

### Что НЕ использовалось

- Декомпиляция мобильного приложения «Салют!» или его firmware
- Дизассемблирование binary
- Извлечение `.proto`-схем или приватных ключей из APK
- Обход технических средств защиты (TLS-bypass, certificate pinning,
  DRM)
- Cracking паролей или brute-force credentials
- Перехват чужого трафика, MITM
- Несанкционированный доступ — пользователь авторизуется собственными
  credentials

## Правовые основания

### Ст. 1280 ГК РФ (императивная норма)

> «Лицо, правомерно владеющее экземпляром программы для ЭВМ или базы
> данных (пользователь), вправе без разрешения автора или иного
> правообладателя и без выплаты дополнительного вознаграждения...
> изучать, исследовать или испытывать функционирование такой
> программы в целях определения идей и принципов, лежащих в основе
> любого элемента программы для ЭВМ»

Также прямо разрешена **адаптация** программ для совместимости с
другими программами и устройствами.

Норма императивна — не может быть ограничена лицензионным договором.
Подтверждено судебной практикой и юридической экспертизой:
- [zakon.ru — Реверс-инжиниринг ПО и российское право](https://zakon.ru/blog/2016/10/11/reversinzhiniring_i_rossijskoe_pravo)
- [blog.lch.legal — Реверс-инжиниринг: правовое регулирование](https://blog.lch.legal/tproduct/388828150-789941865161-revers-inzhiniring-pravovoe-regulirovani)
- [Хакер — Право на реверс](https://xakep.ru/2016/09/02/reverse-rights/)

Правомерное владение учётной записью «Салют!» — на основании договора
оферты с Sber, заключённого при регистрации пользователем своего
аккаунта.

### EU Software Directive 2009/24/EC, ст. 6

В юрисдикциях EU reverse engineering для **interoperability** прямо
разрешён директивой о правовой охране ПО.

### EFF Coders' Rights — Reverse Engineering FAQ

Электронный фронтир документирует устоявшуюся практику: использование
публично доступных endpoint'ов с собственными авторизационными данными
для interoperability не нарушает CFAA/DMCA при отсутствии circumvention
технических средств защиты.

[EFF Coders' Rights Project Reverse Engineering FAQ](https://www.eff.org/issues/coders/reverse-engineering-faq)

### Лицензионная совместимость

Источник CSAFront wire-format — [shuryak/sberdevices](https://github.com/shuryak/sberdevices)
— распространяется под MIT-license, явно допускающей повторное
использование с сохранением copyright notice. Атрибуция приведена в:
- README.md (раздел «Благодарности»);
- модульной docstring `aiosber/auth/csafront.py`;
- CHANGELOG.md для v5.1.0;
- описании GitHub-релиза v5.1.0.

**п. 4.15** содержит два ограничения, которые требуют развёрнутого

Разберём каждое ограничение отдельно.

#### По формулировке «использование объектного/исходного программного кода»

**Программный код в правовом смысле** (ст. 1259 ГК РФ, ст. 1261 ГК РФ)
— это «совокупность данных и команд, представленных в объективной
форме» либо «текст программы», то есть инструкции, исполняемые
процессором / виртуальной машиной (бинарные файлы, Java bytecode,
исходный JavaScript / Kotlin / Swift код мобильного приложения,
firmware колонок).

Проект `ha-sberhome` **не использует и не воспроизводит программный
код Sber**:

- мы не имеем доступа к исходникам и не воспроизводим бинарные файлы
  мобильного приложения «Салют!» или серверов Sber;
- мы не декомпилировали APK, не дизассемблировали firmware;
- мы не используем библиотеки Sber, SDK, .proto-схемы из их кодовой
  базы.

То, что мы воспроизводим — это **wire-format сетевого протокола**
(URI endpoint'ов, структура JSON-тел, имена HTTP-заголовков, порядок
вызовов). Это **спецификация данных на проводе**, а не программный
код. Wire-format и программный код — разные правовые категории:

- код охраняется по ст. 1259 ГК РФ как объект авторского права;
- wire-format охраняется (если охраняется) как know-how / коммерческая
  тайна (ст. 1465 ГК РФ), однако только при соблюдении режима
  коммерческой тайны — введении грифа, ограничении доступа и т.п.

Sber wire-format SMS-OTP flow **не находится в режиме коммерческой
тайны**: соответствующие endpoint'ы ходят через public Internet без
mutual TLS / IP-allowlist'ов, описание flow открыто опубликовано в
проекте [shuryak/sberdevices](https://github.com/shuryak/sberdevices)
с 2023 года и доступно неограниченному кругу лиц.

Дополнительно: даже если толковать «программный код» расширительно,
**ст. 1280 ГК РФ является императивной нормой** и не может быть
ограничена лицензионным соглашением. Изучение протокола ради
совместимости (interoperability) прямо разрешено законом, и условие
оферты в этой части ничтожно (ст. 168 ГК РФ).

#### По формулировке «имитация работы функций Виртуального ассистента»

Виртуальный ассистент «Салют» — это **NLP/voice-сервис**: распознавание
речи (STT), синтез речи (TTS), диалоговая модель, voice-навыки
(SmartApps), запуск smart-сценариев по голосовым командам.

Проект `ha-sberhome` ни одну из этих функций не имитирует:

- **Распознавание речи (STT)** — не делает, аудио не отправляется.
- **Синтез речи (TTS)** — не делает.
- **Диалог / NLP** — не делает.
- **Запуск voice-навыков / SmartApps** — не делает.
- **Голосовые команды для управления устройствами** — не делает.
  Пользователь по-прежнему говорит «Салют, включи свет» в колонку, и
  команда выполняется ассистентом Sber штатным образом. Наш проект
  не подменяет этот канал.
- **Подписка на завершённые события сценариев (для HA-автоматизаций)**
  — это read-only подписка на WebSocket-канал `scenario_widgets`,
  который сообщает «такой-то сценарий пользователя сработал». Аналог
  webhook'а от Sber к HA. Не имитация, а получение уведомлений.

Сам виртуальный ассистент продолжает выполняться на серверах Sber
неизменно — мы не разворачиваем альтернативный voice-сервис, не
встраиваем альтернативный STT/TTS, не маршрутизируем голосовой трафик.

#### Отдельно: `X-Device-ID`, User-Agent, `Referer`, `rsa_data`

Честно обозначим следующий аспект, чтобы у читателя не было ощущения,
что мы что-то скрываем.

Для работы SMS-OTP-flow интеграция передаёт серверу набор технических
полей, без которых endpoint'ы CSAFront отвечают HTML-challenge или
ошибкой:

- **`X-Device-ID`** — рандомно сгенерированный UUID v4. Привязывается
  на стороне сервера к выданной паре токенов. Аналог cookie-сессии для
  REST-клиента.
- **`User-Agent`** — строка вида `Mozilla/5.0 ... Chrome/130.0...`,
  без которой gateway отдаёт страницу-заглушку для «неизвестных
  клиентов».
- **`Referer: SD`** и **`Origin: https://online.sberbank.ru`** —
  заголовки, которые ожидаются веб-форматом CSAFront.
- **`rsa_data`** — словарь со статическими литералами
  (`deviceprint: "{}"`, `htmlinjection: "htmlinjection"`, ...);
  присутствует в схеме CSAFront как anti-bot-фильтр, но сервер не
  валидирует содержимое — только наличие полей.

Можно задаться вопросом: не является ли это «имитацией» легитимного
клиента (мобильного приложения «Салют!» или браузера)?

**Наша квалификация — это не имитация конкретного устройства/клиента
и не обход технических средств защиты, а соблюдение требований
сервера к HTTP-клиенту:**

1. **`X-Device-ID` — собственный UUID, генерируемый клиентом**, а не
   копия идентификатора чужого устройства. Сервер использует его как
   handle сессии (как и любой REST-клиент использует sessionid/cookie).
   Никакого «выдавать себя за устройство X» здесь нет — клиент честно
   объявляет себя новым клиентом.
2. **User-Agent с правдоподобной строкой** — это вынужденная мера для
   совместимости. Стандарт HTTP (RFC 7231 §5.5.3) явно допускает
   произвольное значение User-Agent. Браузеры, mobile-приложения и
   curl/wget всегда передают значение, отражающее их природу. Без
   правдоподобного UA сервер отвергает запрос — это не «обход защиты»,
   а **условие принятия запроса**, аналогично тому, как REST-API
   может требовать заголовок `Accept: application/json`.
3. **`rsa_data` — статические литералы**, не воспроизводящие
   результат работы реального anti-fraud скрипта. Это очередное
   обязательное поле, без которого запрос отвергается. Поскольку
   сервер не валидирует содержимое, никакого «обхода» здесь нет —
   обход подразумевал бы подмену **проверяемых** значений.
4. **Никакого `circumvention` технических средств защиты в смысле
   ст. 1299 ГК РФ / DMCA §1201 не происходит**: мы не дешифруем
   защищённый контент, не обходим signature/license check, не
   подменяем результаты proof-of-work. Мы лишь приводим HTTP-запрос
   в форму, ожидаемую сервером.
5. **Это `interoperability requirement`**, а не имитация. Без этих
   заголовков клиент физически не может взаимодействовать с
   gateway-ом. Сервер сам диктует протокол; клиент его соблюдает.

В судебной практике аналогичные практики (`User-Agent spoofing`,
custom `Referer`, добавление сессионных идентификаторов) **многократно
признавались правомерными** в США и ЕС в контексте interoperability —
наиболее известное дело: **hiQ Labs v. LinkedIn** (US 9th Circuit,
2019/2022), где суд указал, что технические меры, не блокирующие
явно конкретного клиента, не образуют «circumvention».

В РФ прямых прецедентов нет, но ст. 1280 ГК РФ (изучение программ
для обеспечения совместимости) логически распространяется и на
сетевые требования протокола.

**Дополнительно**: проект **не пытается выдать себя за конкретное
устройство пользователя**, не имитирует серийный номер реальной
колонки, не дублирует MAC/IMEI. Клиент представляется как новый
анонимный клиент, авторизующийся под учётной записью пользователя.

Тем не менее, признаём: формулировки оферты можно толковать
расширительно. Поэтому повторно подчёркиваем: **основной (рекомендуемый)
путь авторизации в проекте — публично задокументированный Sber ID
OAuth**, не использующий этих заголовков. SMS-OTP-путь предлагается
как **beta** и помечен соответствующим образом в UI и документации.

#### Отдельная экосистема Sber Smart Home

Sber Smart Home — это **отдельная экосистема устройств** (умные
лампочки, розетки, колонки), для которой Sber ведёт **отдельную
документацию**: [developers.sber.ru/docs/ru/smarthome](https://developers.sber.ru/docs/ru/smarthome).
В ней описана **партнёрская интеграция Cloud-to-Cloud**, прямо
предполагающая участие третьих лиц в обмене данными между Sber и
сторонними системами.

Существование документированной C2C-программы подтверждает, что Sber
**допускает интеграции третьих сторон с экосистемой Smart Home**, и
п. 4.15 Salute Terms нельзя толковать как блокирующий любой обмен с
устройствами умного дома: иначе он противоречил бы собственной
C2C-документации Sber.

#### Резюме по п. 4.15

С учётом изложенного:

- проект не использует и не воспроизводит «программный код» Sber в
  правовом смысле — только wire-format протокола, что является
  отдельной правовой категорией;
- проект не имитирует функции виртуального ассистента — ассистент
  продолжает работать на серверах Sber, мы только подписываемся на
  уведомления о завершении сценариев пользователя;
- технические заголовки (`X-Device-ID`, `User-Agent`, `rsa_data`) —
  собственный UUID и обязательные поля HTTP-протокола, а не
  имитация конкретного устройства пользователя или обход
  технических средств защиты в смысле ст. 1299 ГК РФ;
- ст. 1280 ГК РФ (императивная норма) разрешает изучение протоколов
  ради interoperability независимо от формулировок оферты;
- Sber параллельно ведёт C2C-программу для партнёрских интеграций с
  Smart Home, что подтверждает допустимость существования третьесторонних
  клиентов в этой экосистеме.

Тем не менее, формулировка п. 4.15 широкая, и нельзя исключать
расширительное толкование при правовом конфликте. Поэтому проект
**сознательно ограничивает себя** дополнительными защитными мерами
(см. раздел «Защитные ограничения проекта» ниже) и помечает
SMS-OTP-путь как **beta**, предлагая пользователю как основной путь
**публично задокументированный** Sber ID OAuth flow.

### Документация Sber Smart Home (C2C партнёрская)

Источник: [developers.sber.ru/docs/ru/smarthome/c2c](https://developers.sber.ru/docs/ru/smarthome/c2c)

Cloud-to-Cloud API адресован **производителям умных устройств**,
желающим выставить свои устройства в Sber Smart Home. Описывает
обратное направление (`Sber → vendor backend`), не подходит для
противоположной задачи — управления пользователем своими устройствами
из стороннего клиента.

Документация не содержит запретов на создание альтернативных клиентов
для взаимодействия пользователя с собственными устройствами.

## Защитные ограничения проекта

Проект сознательно ограничивает себя следующим:

- **Только собственные учётные данные пользователя** — исключает
  составы 272/159 УК РФ.
- **Только публичные endpoint'ы gateway-сервера** — без обхода защиты.
- **Один HA-instance — один аккаунт пользователя** — исключает massive
  scraping и аномальную нагрузку.
- **Open-source, некоммерческое распространение** — снимает
  коммерческий конфликт.
- **Атрибуция к источнику wire-format'а (MIT)** — лицензионная
  совместимость.
- **Версионирование, beta-маркировка для нестандартных flow** —
  прозрачность для пользователя.
- **Redaction токенов в diagnostics** — защита приватности.
- **Нет хранения чужих токенов на серверах разработчика** — нет
  посредничества.

## Позиция

Проект `ha-sberhome`:

- Использует **публично задокументированный** Sber ID OAuth flow в
  качестве основного пути авторизации.
- Для **дополнительного (beta) SMS-OTP пути** wire-format взят из
  открытого источника `shuryak/sberdevices` (MIT) с явной атрибуцией;
  никакая декомпиляция, дизассемблирование, обход технических средств
  защиты не использовались.
- Реализует interoperability между устройствами пользователя и Home
  Assistant — целью, прямо предусмотренной ст. 1280 ГК РФ.
- Не имитирует виртуального ассистента «Салют» и не задействует его
  Voice/NLP-функции.
- Не нарушает положений Правил гарантийного обслуживания SberDevices,
  которые прямо допускают использование устройств совместно с
  продуктами третьих лиц.
- Распространяется AS-IS, без гарантий, не аффилирован с ПАО Сбербанк,
  SberDevices или их дочерними структурами. Право собственности на
  товарные знаки Sber, SberDevices, «Салют», SberHome принадлежит
  правообладателям; их упоминание в документации проекта является
  **nominative use** для обозначения совместимости.
